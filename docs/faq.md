# 常见问题

## 这个工具会自动提交成绩到泛雅 / 超星吗？

不会。它只在本地生成评分标准、评阅记录、作品总览图和 Excel 表格。不会在网页上提交、保存、发布、退回或导入成绩。

如果后续需要把分数录入学校平台，应由老师或助教自行检查后手动处理。

## 它是不是自动打分工具？

更准确地说，它是“本地作业评阅辅助工具”。

它可以帮助整理材料、提取 evidence、生成建议分数和学生评语草稿，但评分标准必须由用户确认，草稿评阅也需要用户确认后才会转成正式评阅记录。它不应替代老师的最终判断。

## 学生数据会上传到云端吗？

项目本身按本地文件工作流设计。作业压缩包、临时 evidence、评分标准、评阅记录和导出的 Excel 都保存在本机项目目录中。

使用时仍应注意：如果你把项目文件、输出表格、截图或 demo 上传到 GitHub、网盘或社交平台，就需要先脱敏，避免包含真实学生信息、课程信息和作品版权内容。

## 为什么更适合视觉类和混合作业？

视觉类和混合作业通常会有大量图片、视频、PDF、PPT、源文件和压缩包。传统网页逐个打开会很慢，也很难横向比较全班完成度。

这个工具可以先整理文件、抽帧、渲染 PDF、提取文档内容，并生成作品总览图，适合快速发现整体质量、异常提交、明显高分或低分候选。

## 纯文字作业可以用吗？

可以，但优势不同。

纯文字作业通常不适合依赖作品总览图。更合适的方式是把 docx、pptx、txt、PDF 等文件里的重点文字整理成 `review-text.md`，再按评分标准逐个快速阅读和评阅。

## 压缩包批阅模式和网页批阅模式有什么区别？

压缩包批阅模式适合已经从泛雅 / 超星下载了全班作业总包的情况。它更快、更稳定，也更适合批量生成作品总览图。

网页批阅模式适合没有总包、必须从网页逐个学生进入批阅页的情况。它会更慢，也更依赖浏览器状态和网页加载。

## 为什么一定要先确认评分标准？

不同作业的交付物和评价重点差别很大。例如有的作业重点是最终视频，有的重点是排版设计图，有的重点是报告和过程说明。

因此工具必须先读取或整理作业要求，生成评分标准，再让用户确认。评分标准确认前，不应开始评阅学生。

## 作品总览图可以直接决定最终分数吗？

不建议无条件这样做。

作品总览图适合第一轮快速分档和发现异常。对于边界分、明显高分、明显低分、材料缺失或代表图不充分的学生，应继续查看关键材料。快速批阅会先写入 `draftReviews`，用户确认后才转为正式 `reviews`。

## 评语会不会很模板化？

项目的目标是生成可直接给学生看的评语，而不是后台备注。评语默认应包含作品特点、完成度或具体优点，以及一个可执行的改进方向。

评语不应出现“草稿”“复核”“当前分数”“高分候选”“正式评阅时再看”等内部流程话术。

## 可以用真实学生数据做公开 demo 吗？

不建议。公开 README、截图、演示压缩包、Excel 示例和文章素材都应使用虚构数据。

真实评阅记录和导出的 Excel 可能包含学生姓名、学号、成绩和评语，应按教学资料妥善保存，不要提交到公开仓库。

## 如果我不想用私人账号宣传怎么办？

可以不使用私人微信或小红书。

更低压力的方式是：

- 先只维护 GitHub / Gitee 页面；
- 用虚构 demo 数据完善项目展示；
- 私下发给少量可信老师或助教试用；
- 如果要公开发布，可以使用单独的项目账号；
- 或者只写知乎 / Gitee / GitHub 这类偏文档和搜索的平台内容。
